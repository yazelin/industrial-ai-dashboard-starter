工業 AI Dashboard 入門模板：快速開始

每一步都有：要打的指令 → 真實輸出 → 「成功的話你會看到：…」。照到最後，你會有一個每秒更新的 AGV / 機台監控頁面在瀏覽器上跑。

前置需求

  • Python 3.10 以上（本文以 Python 3.12 實測）
  • Git、會用終端機
  • 不需要任何 API key、token 或 .env。這個 starter 全部用模擬資料，開箱即可跑。

步驟 1：取得程式碼並安裝

git clone https://github.com/yazelin/industrial-ai-dashboard-starter.git
cd industrial-ai-dashboard-starter
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

成功的話你會看到 pip 印出 Successfully installed ...，提示字元前面多了 (.venv)。

步驟 2：啟動服務

一定要在 repo 根目錄（看得到 app/ 的那層）執行，否則會 ModuleNotFoundError: No module named 'app'。

uvicorn app.main:app --reload --port 8000

成功的話你會看到（真實輸出）：

INFO:     Will watch for changes in these directories: ['/.../industrial-ai-dashboard-starter']
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Started reloader process [787819] using WatchFiles
INFO:     Started server process [787821]
INFO:     Waiting for application startup.
INFO:     Application startup complete.

看到 Application startup complete. 就代表服務起來了。終端機會停在這裡持續執行，這是正常的。

步驟 3：打開儀表板

瀏覽器開 http://127.0.0.1:8000/ 。你會看到深藍底的「Industrial AI Dashboard」單頁：最上方一排橘色邊框告警條、中段「AGVs」與「Machines」兩張卡片、最下方一塊每秒在跳的原始 JSON。數字每秒在動就代表 WebSocket 串流通了。

步驟 4：用 curl 驗證 API

curl http://127.0.0.1:8000/api/snapshot

真實輸出（剛好兩台 AGV 電量低於 20，所以 alerts 真的有東西）：

{
  "timestamp": 1780342125.2277737,
  "agvs": [
    {"id": "AGV-1", "x": 47.0, "y": 17.2, "battery": 46.5, "status": "loading"},
    {"id": "AGV-2", "x": 19.0, "y": 21.9, "battery": 11.2, "status": "charging"},
    {"id": "AGV-3", "x": 19.5, "y": 52.4, "battery": 9.3, "status": "charging"}
  ],
  "machines": [
    {"id": "AOI-1", "status": "warn", "ng_rate": 0.52},
    {"id": "PLC-Press-2", "status": "stop", "temperature": 49.2}
  ],
  "alerts": [
    {"level": "warning", "message": "AGV-2 low battery"},
    {"level": "warning", "message": "AGV-3 low battery"}
  ]
}

電量是隨時間擺動的，所以你跑的時候 alerts 可能是空陣列，多刷新幾次就會看到 low battery 告警冒出來。再試 AI summary 佔位 API：

curl http://127.0.0.1:8000/api/ai-summary
{"summary":"3 AGVs online, 2 alerts."}

整體 OK 的判斷標準

  • 終端機停在 Application startup complete.，沒有 traceback。
  • http://127.0.0.1:8000/ 打得開，最下方 JSON 每秒在跳。
  • /api/snapshot 回傳含 agvs / machines / alerts 的 JSON。
  • /api/ai-summary 回傳一句 summary。

常用入口

  • GET /：dashboard UI
  • GET /api/snapshot：目前 AGV / machines / alerts 快照
  • GET /api/ai-summary：AI summary 佔位 API
  • WebSocket /ws：每秒推送即時快照

工業 AI Dashboard 入門模板：架構說明

核心檔案

  • app/main.py：FastAPI、靜態檔案、REST API、WebSocket
  • app/data_source.py：模擬 AGV / machines / alerts 資料
  • static/index.html：dashboard 頁面
  • static/app.js：連線 WebSocket 並渲染資料
  • static/style.css：dashboard UI 樣式

資料流

  1. 使用者或 client 發出請求。
  2. FastAPI / stdio 入口接收資料。
  3. handler 解析訊息與設定。
  4. adapter / tool / search 層執行實際工作。
  5. 回傳最小可理解的結果。

設計原則

  • 先讓流程可跑，再做漂亮抽象。
  • 這個 starter 沒有任何 secret；等你接真實資料來源或 LLM 時，token 才放進環境變數，不要寫死在程式裡。
  • 每一層保持可以替換：資料來源、AI provider、部署方式。
  • 範例程式刻意保持小，方便你看懂後改成自己的版本。

工業 AI Dashboard 入門模板：完整操作流程

quickstart 讓你把東西跑起來；這一節帶你真的改一次資料來源，並在章末做一個我已實跑驗證的小練習。

全貌：資料怎麼從 Python 流到瀏覽器

核心只有一個函式 simulated_snapshot()，它回傳一個 dict，前端只認得這個 dict 的「形狀」。你要做的事，就是把它換成讀真實資料，但保持回傳形狀不變。前端 static/app.js 只依賴：d.agvs（id/x/y/status/battery）、d.machines（id/status）、d.alerts（level/message）。

步驟 1：把模擬來源換成真實來源（before / after）

原則：只改 app/data_source.py，把外部資料 map 成同樣形狀。改之前是用三角函數產生假資料；改成從真實來源讀，例如：

import time, requests   # 記得 pip install requests 並加進 requirements.txt
FLEET_API = "http://10.0.0.5:9000/agvs"   # 換成你的現場端點

def real_snapshot():
    raw = requests.get(FLEET_API, timeout=2).json()
    agvs = [{
        "id": r["agv_name"], "x": r["pos_x"], "y": r["pos_y"],
        "battery": r["soc"], "status": r["mode"],
    } for r in raw["units"]]
    machines = []
    alerts = [{"level": "warning", "message": a["id"]+" low battery"}
              for a in agvs if a["battery"] < 20]
    return {"timestamp": time.time(), "agvs": agvs, "machines": machines, "alerts": alerts}

改完跑出來，/api/snapshot 的 JSON 形狀完全一樣，只是數字變成現場真實值，前端不用改任何程式。

提醒：requests.get 是阻塞呼叫，放在 /ws 迴圈會卡住 event loop；正式接線請用背景任務定時更新一份 cache，handler 只讀 cache。

步驟 2：加入歷史與 AI summary（方向）

  • 歷史：背景任務把每次 snapshot 存進 ring buffer 或時序資料庫，再多開 GET /api/history。
  • AI summary：目前 /api/ai-summary 回傳樣板字串；要接真 LLM 就改 app/main.py 的 summary()。需要你自己的 LLM key，本文不實跑這段；設好金鑰後你會看到 summary 從固定樣板變成模型寫的句子。

動手練習：新增一台機台 + 一條告警規則（已實跑驗證）

目標：在 machines 多一台 CNC-3，並加「溫度 > 80 發 critical 告警」的規則。在 app/data_source.py 的 machines 末端加上：

{"id": "CNC-3", "status": random.choice(["run","run","stop"]), "temperature": round(60+random.random()*30, 1)},

並在 alerts 後面加一行：

alerts += [{"level": "critical", "message": m["id"]+" overheat"} for m in machines if m.get("temperature", 0) > 80]

存檔（--reload 自動重載），刷新 /api/snapshot。這是我實跑擷取的真實輸出（剛好遇到 CNC-3 89.4 度，觸發 critical 告警）：

"machines": [
  {"id": "AOI-1", "status": "run", "ng_rate": 1.54},
  {"id": "PLC-Press-2", "status": "stop", "temperature": 54.1},
  {"id": "CNC-3", "status": "run", "temperature": 89.4}
],
"alerts": [
  {"level": "warning", "message": "AGV-3 low battery"},
  {"level": "critical", "message": "CNC-3 overheat"}
]

瀏覽器的「Machines」卡片會多一列 CNC-3；溫度衝破 80 時，告警條會多一條 critical: CNC-3 overheat。溫度是隨機的，多刷幾次就會看到它出現又消失。這個練習完整體現了 starter 的擴充方式：改 data_source.py 的回傳內容，前端自動跟著變。

完整流程後的下一步

把最小流程跑通、練習做完之後，先別急著堆功能。挑一個你手上真實但很小的情境，讓這個 starter 先把那一個問題解乾淨，再往外長。

工業 AI Dashboard 入門模板：部署筆記

部署前檢查

  • 本機流程已經跑通（/api/snapshot 有回 JSON、儀表板每秒更新）。
  • README 的啟動指令與實際程式一致。
  • 確認 requirements.txt 已被部署平台安裝。
  • 這個 starter 本身沒有 .env、沒有 secret、沒有 webhook，也沒有 /health endpoint，這幾項先不用煩惱。等你接了真實資料來源（可能帶 API key 或內網位址），再把這些秘密放進部署平台的 secrets，不要 commit 進 repo。

啟動指令


uvicorn app.main:app --host 0.0.0.0 --port 8000

常見部署選項

  • Render / Railway / Fly.io：適合快速 demo。
  • VPS + Docker / systemd：適合長期自管。
  • 公司內網主機：適合企業內部工具，但要處理網路與權限。
  • NAS / edge gateway：適合工業或內部自動化場景。

部署後驗證


curl http://127.0.0.1:8000/api/snapshot

接著再測真正的業務流程，不要只看服務有沒有啟動。

實務提醒

部署不是最後一步。正式使用前至少要補：log、錯誤告警、權限控管、備份策略、secret rotation，以及基本監控。

工業 AI Dashboard 入門模板：常見問題與踩雷清單

以下錯誤訊息都是實際跑這個 starter 時會遇到、且實測重現過的。

1. 在錯的目錄啟動：No module named 'app'

uvicorn app.main:app 用 import path 找程式，當前目錄要看得到 app/。在別的目錄啟動會出現：

ModuleNotFoundError: No module named 'app'

修法：回到 repo 根目錄（ls 看得到 app/、static/、requirements.txt）再啟動。

2. 連接埠被占用：address already in use

8000 埠已被別的程式（包括你上一個沒關掉的 uvicorn）占用時：

ERROR:    [Errno 98] error while attempting to bind on address ('127.0.0.1', 8000): address already in use

修法：換埠 --port 8001（網址也跟著改），或 lsof -i :8000 找到 PID 後 kill。

3. WebSocket 連不上：路徑打錯

前端連的是 /ws。若改前端時把路徑打成 /websocket 之類，畫面會停在空白不再更新；連錯路徑伺服器端會回：

server rejected WebSocket connection: HTTP 403

修法：確認 WebSocket URL 路徑是 /ws，協定 http 用 ws、https 用 wss（app.js 已自動判斷）。改完在瀏覽器硬重新整理。

4. 看到空的 alerts，以為壞了

/api/snapshot 有時回 "alerts": [] 不是 bug。告警是「AGV 電量低於 20」才產生，電量隨時間擺動，等幾秒或多刷新就會看到。

5. 終端機卡住不動，其實正常

啟動後游標停在 Application startup complete. 是對的，uvicorn 是常駐服務。另開一個終端機去 curl，要停就在這個視窗按 Ctrl+C。

問別人前準備

  • repo / branch、Python 版本、你在哪個目錄、用什麼指令啟動
  • 完整錯誤訊息（整段 traceback，不是只貼最後一行）
  • 你預期看到什麼、實際看到什麼

工業 AI Dashboard 入門模板：改成你的使用場景

可延伸方向

  • 接 MQTT：背景 task 訂閱 topic，更新最新狀態 cache。
  • 接 ROS2：把 ROS topic bridge 成 dashboard JSON。
  • 接 PLC：用 adapter 讀取 tag，再轉成 agvs / machines / alerts。
  • 加 AI：把異常告警與歷史摘要交給 LLM 生成班報。

改造原則

  • 一次只改一個層次：先改資料來源，再改 UI，再改部署。
  • 先保留原本可跑的範例，另開 branch 做實驗。
  • 每加一個外部服務，就加一個健康檢查或 smoke test。
  • 先做 PoC，再決定要不要產品化。

適合拿來做課程 / 工作坊的題目

  • 從零跑起這個 starter。
  • 改成自己的真實場景。
  • 加入權限、部署與監控。
  • 現場 debug 學員遇到的問題。
